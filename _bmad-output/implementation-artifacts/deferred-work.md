# Deferred Work

Items surfaced during review that are real but not actionable in the story that found them.

## Deferred from: code review of story-1.1 (2026-07-17)

- **`allowScripts` pins exact versions while deps are ranges** — `backend/package.json` ends with `"allowScripts": { "@prisma/engines@7.8.0": true, "prisma@7.8.0": true, "unrs-resolver@1.12.2": true }`, but the dependencies are declared as ranges (`"prisma": "^7.8.0"`). When `^7.8.0` resolves to a patch bump, the allowlist stops matching and npm's install-script approval blocks again — in CI, with no TTY. Surfaces in Story 6.2 (CI lint+test+build gate).
- **Jest `rootDir: "src"` excludes the e2e suite from `npm test`** — `backend/test/app.e2e-spec.ts` lives outside `rootDir`, so only the separate `npm run test:e2e` command runs it. `collectCoverageFrom: ["**/*.(t|j)s"]` also pulls `.spec.ts` files into the coverage denominator. Story 6.2 owns the CI gate and should wire both suites.
- **No graceful shutdown path** — no `app.enableShutdownHooks()`, no `redis.quit()`, no Prisma `$disconnect()` (`PrismaService` implements `OnModuleInit` only), no `connection.close()` on the worker's AMQP handle. SIGTERM is ignored until the 10s SIGKILL. Matters at Epic 6 (production container replacement) and Story 3.3 (in-flight submissions, NFR-04).
- **Two `.env` files hold the same `DATABASE_URL` — consolidate in Story 1.2** — `prisma init` created `backend/.env` (what the Prisma CLI reads via `dotenv/config` in `prisma.config.ts`), while `.env.example` and docker-compose's `env_file:` both live at the repo root. Story 1.1's code review worked around this by pointing `ConfigModule` at both (`envFilePath: ['.env', '../.env']` in `backend/src/app.module.ts`), so nothing is broken — but two sources of truth for one connection string will bite someone. Story 1.2 owns the first migration and touches Prisma config anyway, so it should pick one location and delete the other. Decided by Admin during the Story 1.1 code review (2026-07-17).
- **The create-vite demo landing page is committed as the app** — `frontend/src/App.tsx` (counter, hero image, links to vite.dev/GitHub/Discord/X/Bluesky), ~180 lines of demo CSS in `App.css`, purple accent tokens in `index.css`, `frontend/src/assets/hero.png`, and `<title>frontend</title>` in `index.html`. Not a guardrail breach (Story 1.4 owns the design system) but dead weight Story 1.4 must delete.

## Deferred from: code review of story-1-3 (2026-07-18)

- **ValidationPipe `transform` won't coerce query params to numbers** — `backend/src/common/configure-app.ts:10` registers the global `ValidationPipe` with `transform: true` but no `enableImplicitConversion`. Query params arrive as strings, so a future `?page=&limit=` DTO with numeric fields keeps string values unless each field carries `@Type(() => Number)`. Decide the convention (implicit conversion vs. explicit `@Type`) when Story 2.9 adds the first paginated endpoint, and document it in PROJECT-STANDARDS §6.
- **`isListPayload` structural detection may misfire on domain objects** — `backend/src/common/interceptors/response.interceptor.ts:17` detects the `{ items, meta }` list-payload contract by structural duck-typing (`items` is an array + `meta` is a non-null object). A future controller returning a single domain object that happens to have both fields would be reshaped into `{ data: items, meta }`, dropping its other fields; an array `meta` also passes the check. Story 1.3 deliberately accepted this narrow check since no controller uses the contract yet. Revisit at Story 2.9 (first real paginated endpoint) — consider a branded `PaginatedResult` class checked via `instanceof` instead.

## Deferred from: code review of story-1.4 (2026-07-19)

- **`--color-outline` text is marginally below WCAG AA contrast** — `frontend/src/index.css:34` sets `--color-outline: #727785`, which computes to ~4.2:1 against the surface/background tones it's used on (e.g. Brand's sidebar subtitle, PlaceholderPage's mockup caption) — just under the 4.5:1 AA threshold for normal-size text. Inherited verbatim from `docs/design-system.md`'s frontmatter, not introduced by this diff. Revisit if the design-system source token is ever revised, or if an accessibility pass is scheduled.
- **Type/spacing scale uses raw px, not rem** — `frontend/src/index.css:82` onward declares the entire Inter type scale and 4/8px spacing scale in `px` (e.g. `--text-display: 36px`), weakening browser text-only-zoom accessibility (WCAG 1.4.4). `docs/design-system.md`'s frontmatter itself specifies px (`fontSize: 36px`), so this diff faithfully transcribed the authoritative token source per AC1 — the fix, if any, belongs in the shared design-system doc, not this story.
- **Dark-mode support dropped, no replacement** — the deleted demo CSS carried `color-scheme: light dark` plus a full `prefers-color-scheme: dark` palette; the new token system is light-only. No product requirement calls for dark mode (SRS and design-system.md define light-mode tokens only), and the removed palette was demo-only boilerplate this story's own tasks instruct deleting — but note here in case a future story is asked to add dark-mode support from scratch.

## Deferred from: code review of story-1.7 (2026-07-23)

- **Redis outage takes down all login (fail-closed, no circuit breaker)** — `backend/src/common/guards/login-rate-limit.guard.ts:76` awaits both `hit()` calls with no try/catch; if `client.eval` rejects (Redis down or blipping), the guard throws unhandled and the global filter returns 500 for every login attempt, not just throttled ones — before Story 1.7, a Redis outage did not block issuing an access token. Decided by Admin during the Story 1.7 code review (2026-07-23): defer to Story 2.3, which owns a general circuit-breaker / graceful-degradation mechanism for external-service failures (NFR-11); the login guard should reuse that instead of a one-off fix here.
- **A legitimate shared-NAT student can self-lock their own account from IP-level rejections alone** — `backend/src/common/guards/login-rate-limit.guard.ts:50-76` runs both the IP and account `hit()` calls unconditionally (Task 3's explicit anti-evasion requirement: record both dimensions even when one already tripped). Once the lenient per-IP window (default 30/min) saturates during a class login rush, further IP-rejected attempts still consume the strict per-account window (default 5/min) of the retrying student, which can 429-lock a student who never entered a wrong password — the "don't lock out legitimate students" scenario the story's design notes call load-bearing (NFR-01/05). Decided by Admin during the Story 1.7 code review (2026-07-23): defer, accepted as-is — impact is bounded to a single self-healing 60s window and is the intended trade-off of Task 3's anti-evasion design; revisit only if real exam-time lockouts are observed.
- **`trust proxy=1` is spoofable via a direct `X-Forwarded-For` header until Epic 6's Nginx exists** — `backend/src/main.ts:15` trusts exactly one hop, which is only correct when the sole upstream is a properly-configured Nginx that strips/rewrites inbound `X-Forwarded-For`. `docker-compose.yml` currently publishes the backend's port 3000 directly (`"3000:3000"`) with no `nginx` service in front, so today a direct caller can set their own `X-Forwarded-For` and get a fresh per-IP rate-limit bucket on every request, defeating the per-IP window (the per-account window remains the real, spoof-proof backstop regardless). Explicitly framed in the Story 1.7 spec as prod-correctness for Epic 6. Gate: do not deploy `trust proxy=1` behind an internet-reachable port 3000 before the Epic 6 Nginx layer lands.

## Deferred from: code review of story-1.8 (2026-07-23)

- **No rate limit on `/auth/forgot-password` — email bombing and Resend quota exhaustion** — `backend/src/modules/auth/auth.controller.ts:50-65` adds an unauthenticated endpoint that triggers a third-party email send, with no throttle. A script POSTing one known address 10,000 times floods that victim's inbox, inserts 10,000 rows into `password_reset_tokens`, and burns the Resend free-tier daily quota so *no* legitimate reset email can be delivered for the rest of the day. Story 1.8 correctly complied with its own guardrail — AD-19 names exactly two rate-limited endpoints (login, AI-parse-enqueue) and this is neither — while SRS §9.6 scopes rate limiting more broadly to "endpoint nhạy cảm". Decided by Admin during the Story 1.8 code review (2026-07-23): keep Story 1.8 inside its AD-19 guardrail and open a follow-up hardening story rather than widening scope silently. The infrastructure already exists — `SlidingWindowRateLimiterService` and the two-dimension pattern in `login-rate-limit.guard.ts` — so the follow-up is a guard keyed on email + IP, not new plumbing.
- **`password_reset_tokens` is never purged** — `backend/src/modules/auth/auth.service.ts:184-186` inserts a row per reset request and nothing ever deletes used or expired rows, so the table grows monotonically forever and `tokenHash` lookups plus backups degrade over time. Deferred: a sweep needs a scheduler, and no scheduling mechanism exists in the stack yet (RabbitMQ is planned for AI parsing, not cron). Revisit alongside Epic 6's production concerns, or whenever the first scheduled job lands.
- **Email lookup is case-sensitive, so a capitalised address silently sends nothing** — `backend/src/modules/auth/auth.service.ts:172` does an exact-match `findUnique`. A user registered as `student1@onthi12.local` who types `Student1@onthi12.local` gets `user === null` → no token, no email, and a confirmation screen telling them to check their inbox; they will retry forever with no signal. Deferred, pre-existing: `validateUser` has the identical behavior and Story 1.8's Task 4 explicitly scoped normalization out ("do not introduce email lowercasing here since login doesn't do it either"). Note that `login-rate-limit.guard.ts:103-105` already normalizes case for its Redis key and carries a comment acknowledging the inconsistency. Fix properly in one pass — normalize at every write and read site, or make the column `citext`.
- **The `password_reset_tokens` FK is `ON DELETE RESTRICT`** — `backend/src/prisma/schema.prisma:51`. Consistent with every other relation in the schema, so not a deviation introduced by this story, but for an ephemeral purely-derived token table it means a single leftover row permanently blocks deleting a user; combined with the no-purge item above, every user who has ever requested a reset eventually becomes undeletable. `onDelete: Cascade` is the right semantics for this table specifically. Deferred: no user-deletion feature exists yet — becomes real the moment one ships.
- **Both new auth pages bypass the mandated data-access layer** — `frontend/src/features/auth/forgot-password-page.tsx` and `reset-password-page.tsx` call `apiFetch` inline with hand-rolled `submitting`/`error`/`success` state, while `project-context.md` mandates "Frontend data access goes through TanStack Query hooks + a single API client in `lib/`" and `QueryClientProvider` sits mounted in `main.tsx` unused by any feature. Deferred, pre-existing: this matches the login-page precedent rather than introducing new drift — but it is now three pages deep, so the convention should either be followed repo-wide or amended in `project-context.md`. That call is bigger than one story.
