# Deferred Work

Items surfaced during review that are real but not actionable in the story that found them.

## Deferred from: code review of story-1.1 (2026-07-17)

- **`allowScripts` pins exact versions while deps are ranges** — `backend/package.json` ends with `"allowScripts": { "@prisma/engines@7.8.0": true, "prisma@7.8.0": true, "unrs-resolver@1.12.2": true }`, but the dependencies are declared as ranges (`"prisma": "^7.8.0"`). When `^7.8.0` resolves to a patch bump, the allowlist stops matching and npm's install-script approval blocks again — in CI, with no TTY. Surfaces in Story 6.2 (CI lint+test+build gate).
- **Jest `rootDir: "src"` excludes the e2e suite from `npm test`** — `backend/test/app.e2e-spec.ts` lives outside `rootDir`, so only the separate `npm run test:e2e` command runs it. `collectCoverageFrom: ["**/*.(t|j)s"]` also pulls `.spec.ts` files into the coverage denominator. Story 6.2 owns the CI gate and should wire both suites.
- **No graceful shutdown path** — no `app.enableShutdownHooks()`, no `redis.quit()`, no Prisma `$disconnect()` (`PrismaService` implements `OnModuleInit` only), no `connection.close()` on the worker's AMQP handle. SIGTERM is ignored until the 10s SIGKILL. Matters at Epic 6 (production container replacement) and Story 3.3 (in-flight submissions, NFR-04).
- **Two `.env` files hold the same `DATABASE_URL` — consolidate in Story 1.2** — `prisma init` created `backend/.env` (what the Prisma CLI reads via `dotenv/config` in `prisma.config.ts`), while `.env.example` and docker-compose's `env_file:` both live at the repo root. Story 1.1's code review worked around this by pointing `ConfigModule` at both (`envFilePath: ['.env', '../.env']` in `backend/src/app.module.ts`), so nothing is broken — but two sources of truth for one connection string will bite someone. Story 1.2 owns the first migration and touches Prisma config anyway, so it should pick one location and delete the other. Decided by Admin during the Story 1.1 code review (2026-07-17).
- **The create-vite demo landing page is committed as the app** — `frontend/src/App.tsx` (counter, hero image, links to vite.dev/GitHub/Discord/X/Bluesky), ~180 lines of demo CSS in `App.css`, purple accent tokens in `index.css`, `frontend/src/assets/hero.png`, and `<title>frontend</title>` in `index.html`. Not a guardrail breach (Story 1.4 owns the design system) but dead weight Story 1.4 must delete.

## Deferred from: code review of story-1-3 (2026-07-18)

- **ValidationPipe `transform` won't coerce query params to numbers** — `backend/src/common/configure-app.ts:10` registers the global `ValidationPipe` with `transform: true` but no `enableImplicitConversion`. Query params arrive as strings, so a future `?page=&limit=` DTO with numeric fields keeps string values unless each field carries `@Type(() => Number)`. Decide the convention (implicit conversion vs. explicit `@Type`) when Story 2.9 adds the first paginated endpoint, and document it in PROJECT-STANDARDS §6.
- **`isListPayload` structural detection may misfire on domain objects** — `backend/src/common/interceptors/response.interceptor.ts:17` detects the `{ items, meta }` list-payload contract by structural duck-typing (`items` is an array + `meta` is a non-null object). A future controller returning a single domain object that happens to have both fields would be reshaped into `{ data: items, meta }`, dropping its other fields; an array `meta` also passes the check. Story 1.3 deliberately accepted this narrow check since no controller uses the contract yet. Revisit at Story 2.9 (first real paginated endpoint) — consider a branded `PaginatedResult` class checked via `instanceof` instead.
