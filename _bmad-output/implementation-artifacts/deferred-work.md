## Deferred from: code review of story-1-3 (2026-07-18)

- **ValidationPipe `transform` won't coerce query params to numbers** â€” `backend/src/common/configure-app.ts:10` registers the global `ValidationPipe` with `transform: true` but no `enableImplicitConversion`. Query params arrive as strings, so a future `?page=&limit=` DTO with numeric fields keeps string values unless each field carries `@Type(() => Number)`. Decide the convention (implicit conversion vs. explicit `@Type`) when Story 2.9 adds the first paginated endpoint, and document it in PROJECT-STANDARDS Â§6.
- **`isListPayload` structural detection may misfire on domain objects** â€” `backend/src/common/interceptors/response.interceptor.ts:17` detects the `{ items, meta }` list-payload contract by structural duck-typing (`items` is an array + `meta` is a non-null object). A future controller returning a single domain object that happens to have both fields would be reshaped into `{ data: items, meta }`, dropping its other fields; an array `meta` also passes the check. Story 1.3 deliberately accepted this narrow check since no controller uses the contract yet. Revisit at Story 2.9 (first real paginated endpoint) â€” consider a branded `PaginatedResult` class checked via `instanceof` instead.
