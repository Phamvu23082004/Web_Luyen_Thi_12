- **Redis outage takes down all login (fail-closed, no circuit breaker)** — `backend/src/common/guards/login-rate-limit.guard.ts:76` awaits both `hit()` calls with no try/catch; if `client.eval` rejects (Redis down or blipping), the guard throws unhandled and the global filter returns 500 for every login attempt, not just throttled ones — before Story 1.7, a Redis outage did not block issuing an access token. - **A legitimate shared-NAT student can self-lock their own account from IP-level rejections alone** — `backend/src/common/guards/login-rate-limit.guard.ts:50-76` runs both the IP and account `hit()` calls unconditionally (Task 3's explicit anti-evasion requirement: record both dimensions even when one already tripped). Once the lenient per-IP window (default 30/min) saturates during a class login rush, further IP-rejected attempts still consume the strict per-account window (default 5/min) of the retrying student, which can 429-lock a student who never entered a wrong password — the "don't lock out legitimate students" scenario the story's design notes call load-bearing (NFR-01/05). Decided by Admin during the Story 1.7 code review (2026-07-23): defer, accepted as-is — impact is bounded to a single self-healing 60s window and is the intended trade-off of Task 3's anti-evasion design; revisit only if real exam-time lockouts are observed.

Explicitly framed in the Story 1.7 spec as prod-correctness for Epic 6. Gate: do not deploy `trust proxy=1` behind an internet-reachable port 3000 before the Epic 6 Nginx layer lands.

## Deferred from: code review of story-1.8 (2026-07-23)

- **No rate limit on `/auth/forgot-password` — email bombing and Resend quota exhaustion** — `backend/src/modules/auth/auth.controller.ts:50-65` adds an unauthenticated endpoint that triggers a third-party email send, with no throttle. A script POSTing one known address 10,000 times floods that victim's inbox, inserts 10,000 rows into `password_reset_tokens`, and burns the Resend free-tier daily quota so *no* legitimate reset email can be delivered for the rest of the day. Story 1.8 correctly complied with its own guardrail — AD-19 names exactly two rate-limited endpoints (login, AI-parse-enqueue) and this is neither — while SRS §9.6 scopes rate limiting more broadly to "endpoint nhạy cảm". Decided by Admin during the Story 1.8 code review (2026-07-23): keep Story 1.8 inside its AD-19 guardrail and open a follow-up hardening story rather than widening scope silently. The infrastructure already exists — `SlidingWindowRateLimiterService` and the two-dimension pattern in `login-rate-limit.guard.ts` — so the follow-up is a guard keyed on email + IP, not new plumbing.
- **`password_reset_tokens` is never purged** — `backend/src/modules/auth/auth.service.ts:184-186` inserts a row per reset request and nothing ever deletes used or expired rows, so the table grows monotonically forever and `tokenHash` lookups plus backups degrade over time. Deferred: a sweep needs a scheduler, and no scheduling mechanism exists in the stack yet (RabbitMQ is planned for AI parsing, not cron). Revisit alongside Epic 6's production concerns, or whenever the first scheduled job lands.
- **Email lookup is case-sensitive, so a capitalised address silently sends nothing** — `backend/src/modules/auth/auth.service.ts:172` does an exact-match `findUnique`. A user registered as `student1@onthi12.local` who types `Student1@onthi12.local` gets `user === null` → no token, no email, and a confirmation screen telling them to check their inbox; they will retry forever with no signal. Deferred, pre-existing: `validateUser` has the identical behavior and Story 1.8's Task 4 explicitly scoped normalization out ("do not introduce email lowercasing here since login doesn't do it either"). Note that `login-rate-limit.guard.ts:103-105` already normalizes case for its Redis key and carries a comment acknowledging the inconsistency. Fix properly in one pass — normalize at every write and read site, or make the column `citext`.
- **The `password_reset_tokens` FK is `ON DELETE RESTRICT`** — `backend/src/prisma/schema.prisma:51`. Consistent with every other relation in the schema, so not a deviation introduced by this story, but for an ephemeral purely-derived token table it means a single leftover row permanently blocks deleting a user; combined with the no-purge item above, every user who has ever requested a reset eventually becomes undeletable. `onDelete: Cascade` is the right semantics for this table specifically. Deferred: no user-deletion feature exists yet — becomes real the moment one ships.
- **Both new auth pages bypass the mandated data-access layer** — `frontend/src/features/auth/forgot-password-page.tsx` and `reset-password-page.tsx` call `apiFetch` inline with hand-rolled `submitting`/`error`/`success` state, while `project-context.md` mandates "Frontend data access goes through TanStack Query hooks + a single API client in `lib/`" and `QueryClientProvider` sits mounted in `main.tsx` unused by any feature. Deferred, pre-existing: this matches the login-page precedent rather than introducing new drift — but it is now three pages deep, so the convention should either be followed repo-wide or amended in `project-context.md`. That call is bigger than one story. **RESOLVED 2026-07-23 by Admin — the rule was amended, not the code.** `project-context.md` and `ARCHITECTURE-SPINE.md` now scope TanStack Query to server state (`useQuery` for every GET, `useMutation` for every write that invalidates cache) and grant a narrow exception to pre-auth forms that have nothing cached to read or invalidate — which is exactly what these three pages are. The three auth pages stay as-is. No new exceptions from Epic 2 onward: the parsing-status poll (Story 2.2) and the question edit/confirm writes (Stories 2.4/2.5/2.7) must use hooks.

## Deferred from: manual smoke run of story-2.2 (2026-07-24)

Surfaced by parsing a **real** exam — the official 2026 THPT Toán paper (mã đề 0101, 12 pages: 4 pages of exam + 8 pages of a bundled solution guide). All three items are product/architecture decisions, not defects in Story 2.2: every one of its four ACs holds on this input, and the system behaved correctly and safely throughout.

- **⚠️ The v1.1 question model covers only 12 of the 22 questions on a real exam paper (55%).** The 2026 THPT paper has three parts: **Phần I** — 12 × four-option single-answer (supported); **Phần II** — 4 questions, each with four independent Đúng/Sai sub-statements a)-d) (**not supported**); **Phần III** — 6 short-answer questions where the student enters a number (**not supported**). Gemini correctly skipped parts II and III: `gemini-question-extractor.ts`'s prompt rule 3 says a question must have exactly four options keyed A-D or be skipped, and `responseSchema` hard-codes that shape. Verified in the smoke run — page 1 → 10 questions, page 2 → 2 questions, pages 3-12 → 0.

  **The SRS's stated reason for excluding these does not apply to them.** §1.4 excludes "chấm điểm tự động cho câu hỏi tự luận" because it "yêu cầu xử lý ngôn ngữ tự nhiên phức tạp". Neither part is tự luận and neither needs NLP: Phần II is four booleans per question, Phần III is a numeric equality check. Both are mechanically auto-gradable. §2.3's "chỉ gồm câu hỏi trắc nghiệm một đáp án đúng" appears to reflect an assumption about exam format that the 2025+ THPT structure has outgrown, rather than a grading-feasibility limit.

  Cost if adopted — this is **epic-level, not story-level**: `questions` needs an answer-format discriminator (note this is distinct from the topic/"dạng bài" tag SRS §7 rules out — that exclusion is about subject tagging, not answer shape); `correct_answer String?` cannot hold Phần II's four booleans or Phần III's numeric value; `options` JSON is shaped `{A,B,C,D}` and Phần III has no options at all; **Story 3.3's grading gains partial credit** (Phần II awards different marks for 1/2/3/4 correct sub-statements — no longer a binary match, which is a change to the merge-blocking NFR-04 path); Stories 2.4/2.5 need per-type confirmation widgets and Story 3.4 per-type input widgets; and 2.2's prompt + `responseSchema` need a per-type contract. Touches Epic 2 **and** Epic 3.

  **✅ RESOLVED 2026-07-24 — adopted, not deferred.** Admin's first call was to park this for the Epic 2 retrospective; on reflection the same day, Admin chose to fix it immediately instead, because the cost only rises: `ParsedQuestion` is the cross-module contract Story 2.2's own Dev Notes single out ("get the shape right here rather than reshaping it four stories later"), and exactly **one** story depends on it today versus four after 2.4/2.5/2.7 — and after Story 3.3 the change would reopen the transactional grading core that NFR-04's merge-blocking tests guard.

  The planning path was followed rather than amending an in-flight story: **SRS raised to v1.2** (new §3.6 with QTYPE-01..06 defining the three types, the scoring scale, and the data shapes; plus §1.2/§1.4/§2.3/EXAM-06/TAKE-02/TAKE-04/TAKE-05/§7/§10 amended), **PRD amended** (Glossary gains Question Type / Statement / Partial Credit; FR-5, FR-7, FR-16, FR-17 updated), and **Story 2.2b** added to `epics.md` immediately after 2.2 and before 2.3. Story 2.2 itself ships unchanged — all four of its ACs hold. Evidence PDF: `de-chinh-thuc-ky-thi-tot-nghiep-thpt-nam-2026-mon-toan.pdf`.

  **Scoring rules supplied by Admin, 2026-07-24 — the earlier "unverified scale" warning is resolved.** Four subject groups, each totalling 10: Toán 22 questions (12/4/6, Phần III at 0,5); Vật lí/Hoá học/Sinh học 28 (18/4/6, Phần III at 0,25); Lịch sử/Địa lí/GDKT&PL 28 (24/4, **no Phần III**); Tiếng Anh 40 (all Phần I). The structure collapses neatly: **`mcq_single` is 0,25 and `true_false_group` is max 1,0 in every subject — only `short_answer` varies (0,5 Toán vs 0,25 for the three sciences).** The partial-credit scale is **non-linear** (1→0,1 · 2→0,25 · 3→0,5 · 4→1,0); computing it as `correct ÷ 4` would pay 0,5 for two correct instead of 0,25.

Existing rows hold only `Toan`/`Toán`, both mapping to `toan`, and no frontend code references the field yet — so this is a backend-only change with no real data at risk.

That makes the raw maximum always exactly 10, so **there is no score-normalization step anywhere in the system**, and it makes `short_answer` structurally impossible for the subjects that have no Phần III. Both former open questions are now unreachable states rather than unanswered ones.
