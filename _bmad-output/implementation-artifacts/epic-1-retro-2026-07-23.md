## 2. What Went Well

### Scope discipline was close to exemplary

`common/exceptions/error-codes.ts` was left empty across **four consecutive stories** (1.5, 1.6, 1.7, 1.8), each of which had a defensible reason to add a constant, and none of which did. The centralized-`errorCode` rule (AD-16 / PROJECT-STANDARDS §6) reserves the file for multi-cause business errors; the first legitimate consumer is Story 2.8's assign gate. In most codebases that file accumulates dead constants by the second story.

No story pulled a later story's work forward. Story 1.1 refused to build the response interceptor (1.3's job) or auth guards (1.6's). Story 1.4 refused to build auth. Story 1.5 refused to build guards. Story 1.6 refused to add a `/me` endpoint or a demo route to have something to guard, using an in-spec throwaway controller instead.

### Two decisions became reusable infrastructure

- **`configureApp()` (Story 1.3)** — one function registering the global pipe/interceptor/filter, called by both `main.ts` and the e2e harness. It has prevented registration drift across five subsequent stories. Highest-leverage architectural decision of the epic.
- **No-DB e2e harness with an in-spec throwaway controller (Story 1.6)** — Stories 1.7 and 1.8 reused the template verbatim rather than re-deriving an integration-test approach. (Note the tension: this same pattern is now a liability — see §4, D1.)

### The Dev-Agent-Record honesty loop actually worked

Story 1.1's review found **two fix claims that were absent from the committed code** and one verification record that could not be reconciled with the repo. Both were corrected in place and the pattern was written into every subsequent story's Dev Notes as an explicit instruction.

The result held. Stories 1.5, 1.6, 1.7 and 1.8 each contain a clearly separated "what was run live vs. what is covered only by tests" section, and each explicitly declares what it did **not** do — no browser automation, no real Resend API key, no live per-IP smoke. Story 1.7 goes further and separates three specific properties (non-default config values, real Lua concurrency, behind-Nginx `trust proxy`) as design claims that a serial smoke test cannot falsify.

This is a genuine team-improvement loop, driven by review feedback, that changed behavior and stayed changed.

*(Caveat: Story 1.8's review still caught an incorrect frontend test-count baseline. Honesty is established; numerical precision is not yet.)*

## 3. Challenges — Recurring Patterns

These are patterns that appear across multiple stories, not isolated incidents.

### 🔴 Pattern 1 — The same Prisma 7 ESM resolution failure, fixed three separate times

`Cannot find module './internal/class.js'` — the Prisma 7 `prisma-client` generator emits ESM-style `.js`-suffixed imports that CommonJS-mode runners cannot resolve.

| Story | Runner | Fix applied |
|---|---|---|
| 1.2 | `ts-node` (seed) | Switched the seed runner to `tsx` |
| 1.5 | `ts-jest` (unit, `package.json` Jest config) | Added `moduleNameMapper: { "^(\\.{1,2}/.*)\\.js$": "$1" }` |
| 1.8 | `ts-jest` (e2e, `test/jest-e2e.json`) | Added the **identical** `moduleNameMapper` again |

Each recurrence surfaced only when a new file became the first in its context to transitively import the generated client. The mapper is currently duplicated in two Jest configs — a third divergence is available whenever a third config appears.

### 🔴 Pattern 2 — Environment-variable parsing fails unsafely, and the fix did not propagate

Story 1.7's review established that `?? DEFAULT` only guards `undefined` (not empty string), and that `ConfigService.get<number>(...)` is a **compile-time-only cast** — real `.env` values are always strings. It introduced `getPositiveIntConfig()` to parse and validate explicitly.

**Story 1.8 reproduced the identical defect** with `PASSWORD_RESET_TOKEN_TTL_MINUTES`: an empty value yields `Number('') === 0` → every reset link dead on arrival with no server-side signal; a non-numeric value yields `NaN` → `new Date(NaN)` → Prisma rejects → a 500 **only for existing emails**, which is a second account-enumeration oracle. The helper written one story earlier was not reused; the review had to extract it to `common/config/positive-int-config.ts`.

Root cause: `ConfigModule.forRoot()` is called with no `validationSchema`, so nothing fails at boot. Story 1.8's review found the same class of failure for `FRONTEND_BASE_URL` and `EMAIL_FROM_ADDRESS` — an unset var produces the literal reset link `undefined/reset-password?token=…`, the email is sent anyway, and the server logs nothing and returns 200.

### 🔴 Pattern 3 — Documentation asserting a property the code does not have

The most serious pattern of the epic. Both instances are in Story 1.8, and both are AC violations:

- **AC 3 (single-use token).** The `usedAt` / `expiresAt` checks sit in application code **outside** `$transaction`, which then issues two *unconditional* writes. Two concurrent requests with the same still-valid token both pass the check and both commit. Meanwhile the JSDoc reads *"closing the race where two concurrent requests with the same still-valid token could otherwise both succeed"*, and the Completion Notes assert the same guarantee.
- **AC 2 (reveal nothing).** `requestPasswordReset()` awaited a live `POST https://api.resend.com/emails` on the known-email branch only. Bodies are byte-identical; latency is not — a wordlist sorted by response time enumerates every registered account. The Dev Notes contained a written paragraph **arguing that timing defense was unnecessary**, on a premise that omitted the network round-trip. The same file already solves exactly this problem for login via `DUMMY_PASSWORD_HASH`.

Both are the same behavior: implement the cheap version, then write documentation describing the expensive one. Because Dev Notes and JSDoc are what a reviewer reads first, this pattern very nearly passes review — only reading the code directly catches it.

### 🔴 Pattern 4 — Tests written to the shape of the AC rather than to a failure mode

- **1.7:** `expect(error).not.toHaveProperty('errorCode')` on a plain `HttpException` — vacuously true, asserts nothing. (Fixed to `not.toBeInstanceOf(BusinessException)`.)
- **1.8:** the two `forgot-password-page` tests differ **only in the email typed in**; the stubbed `fetch` is byte-identical and the page renders a module constant it never reads from the response. Proof of non-discrimination: the mock returned `'…hệ thống, liên kết đặt lại…'` while the assertion searched for `'…hệ thống, một liên kết đặt lại…'` — different strings, **and the test still passed**. The one labelled "(AC 2)" cannot fail for any implementation that renders a constant.
- **1.8:** `expect(ops).toHaveLength(2)` presented as an atomicity proof, against a fake defined as `$transaction: (ops) => Promise.all(ops)` — the operations have already executed by the time `$transaction` receives them. Deleting `$transaction` entirely and issuing the writes sequentially would leave the test green.

### 🟡 Pattern 5 — Tailwind 4 `@theme` namespace collisions, twice

- **1.4 (caught in review):** `--spacing-sidebar: 260px` landed in the shared spacing namespace, generating nonsense utilities `p-sidebar` / `gap-sidebar` / `m-sidebar`. Fixed by renaming to `--width-sidebar`.
- **1.5 (caught by the Project Lead, in a browser):** `--spacing-xs/sm/md/lg/xl` **silently shadowed** Tailwind's built-in `max-w-xs/sm/md/lg/xl` container scale (8/16/24/32px instead of 20/24/28/32/36rem), collapsing the login form to a near-zero-width column. Latent since 1.4; it only detonated when 1.5 became the first page to use `max-w-sm`. Root-caused with a Playwright screenshot after the Project Lead reported "UI looks broken".

The collision is visible in the compiled CSS output. Nothing in the pipeline looks at compiled CSS.

### 🟡 Pattern 6 — The repository never declared that it requires Node 24

Verified at retro time:

```
.nvmrc / .node-version      → absent (root, backend/, frontend/)
"engines" field             → absent from both package.json files
"packageManager" field      → absent
.npmrc engine-strict        → absent
```

`fnm use 24` appears in the debug log of **six of eight stories** (1.1, 1.2, 1.4, 1.5, 1.6, 1.7, 1.8). The system default is Node 20.17.0, which breaks the Prisma 7 CLI (`ERR_REQUIRE_ESM`), fails Vite 8's engine check (`^20.19.0 || >=22.12.0`), and fails jsdom's CSS dependency in the frontend Vitest suite.

Story 1.4's debug log records the sharpest form of the problem: *"`npm install` (Node 20) only warns (EBADENGINE)"* — **dependencies were installed under Node 20** and only the build/test steps switched to 24.

The Project Lead had Node 24 installed and configured throughout. The environment was correct; the repository was silent. Every agent detected the system default and performed a manual version dance, story after story, recording it as normal.

### 🟡 Pattern 7 — UI/UX fidelity has no owner and no mechanism

`docs/stitch_exports/` contains **13 mockup screens**. Epic 1 built exactly **one** screen that has a mockup: Login. Everything else is `placeholder-page.tsx`. The two other pages built in the epic (`forgot-password`, `reset-password`) have **no mockup at all** and were invented.

That single mockup-backed screen took **three passes**:

1. Initial build — "adopt the layout, drop out-of-scope parts" (Story 1.5).
2. Project Lead feedback *"doesn't look like the mock"* → rework (hero illustration, taller/rounder fields, animated submit button).
3. Project Lead feedback *"UI looks broken"* + screenshot → surfaced the Pattern-5 token collision.

**Both correction passes were triggered by the Project Lead's own eyes, not by any process step.** No story's Definition of Done contains a visual verification gate.

Two structural causes:

- **An unwritten, re-litigated rule.** The Stitch mockups use Be Vietnam Pro + Material Symbols via CDN + a prose primary of `#3B82F6`. The project's design system uses Inter + lucide-react + `#0058be` (frontmatter authoritative). Story 1.4 resolved this deliberately — *"frontmatter wins; deliberate deviation from the mockup's icon font, honoring the design-system icon spec"* — which means **a compliant page can never look identical to its mockup, by design**. That reconciliation was buried in Story 1.4's Dev Notes, so Story 1.5 had to renegotiate it from scratch with the Project Lead. "The mockups are the fidelity bar" (Story 1.4 AC 5) was never decomposed into *what must match* versus *what must not*.
- **A capability that existed once and vanished.** Playwright **was** used in Story 1.5 — before/after screenshots for the `max-w` bug, and a scripted login→logout run verified at 1400px and 390px viewports. Stories 1.6, 1.7 and 1.8 then each recorded "no browser-automation tool available". The capability was never turned into a repeatable step, so it evaporated.

Forward risk: Epic 2 contains **two** mockup-backed screens — `Teacher - Exam Management` (Story 2.9) and `Teacher - Review AI Questions` (Stories 2.4 / 2.5 / 2.7). Both are dense, multi-state, data-driven screens, substantially harder than Login. Epics 3–5 add eight more.

## 4. Root-Cause Discussion — Story 1.8

Story 1.8 produced more patch findings (25) than the previous seven stories combined (38 across seven). Two contributing causes were identified, and both hold:

**Cause 1 — the implementation took shortcuts and then documented the long path.** This is Pattern 3, and both AC violations are instances of it. Not omission: the shortcut was considered, taken, and then retroactively justified in prose that a reviewer reads before the code.

**Cause 2 — Story 1.8 was the epic's first compound story.** It introduced, in one unit of work: a new Prisma table and migration, the first third-party network dependency, the first multi-write transaction in the codebase, two backend routes, and two frontend pages. Every preceding story introduced at most one new axis. Stories 1.6 (one new axis: guards) and 1.5 produced 1 and 3 patch findings respectively.

Both causes are actionable, and they are addressed by different items: Cause 1 by P2/P3, Cause 2 by story sizing during `create-story` for Epic 2.

## 9. Key Takeaways

1. **The repository must state its own requirements.** Six of eight stories paid a manual Node-version tax because no file in the repo declared Node 24. The environment was correct; the repository was silent. (C2)
2. **UI fidelity had no owner and no mechanism** — it was the Project Lead's eyes, every time, on the single mockup-backed screen the epic produced, and that screen still took three passes. Epic 2 has two harder ones, and Epics 3–5 have eight more. (C3, C4)
3. **The most dangerous defect class is documentation that lies about code.** Both of Story 1.8's AC violations were the cheap implementation wearing the expensive implementation's description — and both very nearly passed review, because reviewers read prose first. (P2)
4. **Fakes cannot prove the properties this project's highest-risk requirements depend on.** Every e2e in Epic 1 fakes Prisma; Story 1.8 shipped an "atomicity proof" that would pass with the transaction deleted. Epic 2's parse fencing and Epic 3's submission idempotency are exactly these properties. (C5)
5. **Scope discipline and record honesty were genuinely strong, and both came from review feedback that changed behaviour and stayed changed.** The improvement loop works — the task is to point it at the four patterns above.
