# 03 — Code Review Process

## Principle

Code review exists to catch defects, share knowledge, and maintain consistency — not to gatekeep or demonstrate superiority. Reviews should be prompt, constructive, and focused on what matters.

## Default Process

### Who Reviews

| Project Size | Requirement |
|-------------|-------------|
| Solo developer (AI-assisted) | Self-review after AI generates code; use BMad adversarial review for critical modules |
| 2-3 developers | At least 1 peer review before merge |
| 4+ developers | At least 1 peer review; security-sensitive changes require senior/lead review |

### AI Review Layers (precede human review)

Human review is the decision; AI layers are the pre-filter. On every PR, before a human is asked to approve:

1. **QA-agent adversarial pass** (BMAD) — surfaces defects, edge cases, and missing tests.
2. **AI PR reviewer** — reviews the full diff in a sandbox and posts inline findings.

The author resolves machine findings first, so human reviewers spend their attention on judgment calls — architecture, trade-offs, business correctness — not on what a tool can catch. **A human approver (≥1) is still required and non-negotiable;** AI findings are input, not a verdict.

### PR Requirements Before Review

1. **CI passes** — lint, typecheck, and tests are green
2. **PR description** explains what and why (not just "fix bug")
3. **Scope is reasonable** — PRs over 400 lines of meaningful change should be split unless they're schema migrations or generated code
4. **Self-reviewed** — the author has read their own diff before requesting review

### Review Turnaround

- **Target**: Review within 4 business hours of request
- **Maximum**: Review within 1 business day
- **Blocked?**: If a reviewer can't review in time, they reassign — don't let PRs sit

### What Reviewers Should Check

**Always check (every PR):**

1. **Does it do what the PR description says?** — Read the description, then the code. Do they match?
2. **Business logic in the right layer?** — Validation in validators, business rules in services, data access in repositories, HTTP concerns in controllers only.
3. **Error handling** — Are errors thrown as typed AppErrors with error codes? Are edge cases handled?
4. **Security** — No secrets in code, no raw SQL with user input, proper auth checks on protected endpoints.
5. **Naming** — Are functions, variables, and files named clearly and consistently with the project conventions?

**Check for critical modules (auth, balance, state machine):**

6. **Test coverage** — Are the must-have test cases covered? Are edge cases tested?
7. **Concurrency safety** — Does code that touches shared state (balances, approvals) use proper locking?
8. **Audit trail** — Are business operations logged to the audit system?

**Don't waste time on:**
- Formatting (ESLint + Prettier handle this)
- Stylistic preferences that aren't documented conventions
- Hypothetical future requirements ("what if we need to support X later?")

### Review Comments

**Be specific:**
- Bad: "This doesn't look right"
- Good: "This reads the balance without SELECT FOR UPDATE — concurrent requests could see stale data. See the pattern in leave-balance.service.ts:42"

**Categorize your comments:**
- `[blocking]` — Must fix before merge. Used for bugs, security issues, data integrity risks.
- `[suggestion]` — Take it or leave it. Used for style preferences, alternative approaches.
- `[question]` — Seeking understanding. Used when the reviewer wants to learn the reasoning.
- `[nit]` — Trivial. Typo, minor naming preference. Fix if easy, skip if not.

Only `[blocking]` comments prevent merge approval.

### Review Etiquette

1. **Review the code, not the person** — "This function doesn't handle null" not "You forgot to handle null"
2. **Acknowledge good work** — A quick "clean approach here" builds team morale
3. **Don't rewrite** — If you'd write it differently but their approach works correctly, that's a `[suggestion]`, not a `[blocking]`
4. **Assume good intent** — If something looks wrong, ask before assuming it's a mistake
5. **Respond to all comments** — Authors should respond to every review comment, even if just "done" or "won't fix because..."

### AI-Generated Code Reviews

When code is generated by AI (Claude Code):
- **The author is still responsible** — AI-generated code needs the same review as hand-written code
- **Watch for AI patterns** — Over-abstraction, unnecessary error handling, verbose comments, phantom imports

### AI-Assisted Code Review Tools

AI review tools supplement — never replace — human review. Use them **before** creating a PR (developer self-check) and **at quality gates** (critical modules).

#### Available Tools

| Tool | What It Does | How to Use | Platform |
|------|-------------|------------|----------|
| **Claude Code `/code-review`** | 4 parallel agents: CLAUDE.md compliance (x2), bug detection, git history analysis. Confidence scoring (80+ threshold). | Run `/code-review` in Claude Code on your branch | GitHub only (requires `gh` CLI) |
| **PR Review Toolkit agents** | 6 specialized agents: comment accuracy, test coverage, silent failures, type design, general review, code simplification | Ask Claude Code naturally: "check test coverage", "review error handling", "simplify this code" | Any platform (runs on local code) |
| **BMad Adversarial Review** | Critical review of code or documents. Attitude-driven: finds flaws, contradictions, security gaps. | `/bmad-review-adversarial-general` | Any platform |
| **BMad Edge Case Hunter** | Exhaustive path analysis. Method-driven: walks every branch and boundary. Orthogonal to adversarial review. | `/bmad-review-edge-case-hunter` | Any platform |

#### Platform Compatibility

Claude Code's `/code-review` plugin and its `--comment` flag (auto-posting review to PR) currently require **GitHub**. For Azure DevOps projects (our default), use this approach instead:

**Pre-PR developer self-check (any platform):**
```
# In Claude Code, on your feature branch:
"Review my recent changes for bugs"           → triggers code-reviewer agent
"Check test coverage for this PR"              → triggers pr-test-analyzer agent
"Check for silent failures in error handling"  → triggers silent-failure-hunter agent
"Simplify this code"                           → triggers code-simplifier agent
```

**Quality gates for critical modules (any platform):**
```
/bmad-review-adversarial-general    → critical review of code
/bmad-review-edge-case-hunter       → boundary and path analysis
```

#### Recommended AI Review Workflow

```
Developer writes code
    │
    ├── Ask Claude Code: "review my changes"     ← PR Review Toolkit (local)
    ├── Fix issues found
    │
    ├── If auth/security module:
    │   └── /bmad-review-adversarial-general     ← BMad (quality gate)
    │
    ├── If business logic (state machine, calculations):
    │   └── /bmad-review-edge-case-hunter        ← BMad (quality gate)
    │
    ├── Create PR
    │
    └── Human peer review                        ← Always required
```



#### Future: CI/CD Integration

When Claude Code `/code-review` adds Azure DevOps support, it can be integrated into Azure Pipelines as an automated PR review step. Until then, AI review is a **local developer workflow**, not a CI pipeline step. Monitor Anthropic's releases for platform updates.


### Merge Approval

- At least 1 approval required (configurable per project in Tier 2)
- All `[blocking]` comments resolved
- CI passes
- No merge conflicts with target branch
- Author merges their own PR (they know when it's ready)
